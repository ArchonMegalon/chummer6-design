# Public help copy

## Public feedback lane

If you want to help without touching premium participation at all, use the public front door:

* file a public bug
* send lightweight product feedback
* flag confusing public copy
* suggest a future lane
* point at a horizon that sounds more useful than the current list

That keeps feedback public and legible instead of making normal humans spelunk through design canon.

## Private crash lane

If Chummer closes unexpectedly, the first safe path is a private in-product crash report:

* review the redacted diagnostics bundle
* send it privately
* or keep the full bundle local while only the redacted envelope reaches Hub

Private crash diagnostics should not be forced into a public issue or a chat widget.

## Guest versus linked copy

Public help copy may offer both:

* download and try Chummer now
* link this copy to your account so we can keep your reports, suggest the right updates, and tell you when your issue is fixed

That is a relationship choice, not a special per-user build.

## Closure rule

Help copy must not imply that "merged" means "fixed for you."

The user-facing promise is:

* we received your report
* we may ask for one more detail
* we may point you at a known issue
* we tell you when the fix is actually available on your channel
* we may ask later whether that solved it

## Guided contribution lane

Guided contribution is the bounded "I want to help with real execution" lane.

It is:

* opt-in
* temporary
* additive on top of the free baseline
* still subject to review before anything lands

Public copy should prefer `participate` and `guided contribution`.
Operator phrasing belongs behind the scenes.

## Privacy and review safety

Public participation copy must say all of this plainly:

* the cheap baseline remains the default path
* guided contribution is not a merge bypass
* final landing still goes through review
* people can stop or revoke later
* private recognition settings remain valid even when badges or leaderboards exist

## Free later note

Some expensive future lanes may begin as **guided previews** while approvals, compatibility, editorial work, provenance, or support cost remain unusually heavy.

That is not the same thing as a permanent paywall.
When those lanes become cheaper and safer to operate broadly, the long-run intent is wider access rather than keeping the interesting parts locked forever.

## Start with the product surface, not the repo trail

The hosted front door should answer the normal customer questions directly. Use Help, FAQ, Downloads, Account, and Contact before you go hunting through implementation repos.

- Downloads explains the current preview, installer, and update posture.
- Account handles sign-in, recovery, devices, and tracked support.
- Contact opens a first-party support case without a GitHub account.
- FAQ answers the normal preview and product questions in plain language.

## Use the release page and install help as one flow

Start with the recommended installer and the release page. If the path is unclear, use install help and known issues instead of guessing between package types or runtime heads.

- One recommended installer is the default path.
- Alternative heads and manual packages are advanced paths.
- Account-aware downloads keep the install handoff attached from first launch.
- Devices and access is where linked copies and claim codes stay visible later.

## Product help should become a support case, not a side-channel rumor

The first-party intake is the default lane for install trouble, product bugs, and practical feedback. GitHub is the advanced public issue lane, not the main customer entry point.

- Use Contact for install trouble, product bugs, or UX friction.
- Use Account > Support when you want tracked case history in the signed-in shell.
- Use GitHub only when you intentionally want a public technical issue trail.

## Participation is optional and separate from support

Guided contribution is not the price of admission. Support and product use should stand on their own before anybody decides to lend deeper help.

- You do not need to participate to report a bug or ask for help.
- Public feedback stays separate from guided contribution.
- Recognition and contribution history should not outrun customer basics.
